# Fully Agentic Interview Architecture

## Purpose

This document defines how to evolve the current interview system into a fully agentic interview engine.

In the target design, the interview agent should:

- start with full session context before asking the first question
- ask questions grounded in JD, resume, company context, and extracted skill map
- decide whether to probe deeper, verify resume ownership, switch topics, or end the interview
- maintain explicit interview state across turns
- generate structured thread-level and final evaluation outputs

This document is intended to be the implementation contract for the backend work that follows.

---

## Current Reality

The current backend has useful building blocks:

- LangGraph-based interview loop
- tool calling
- session persistence
- JD/resume extraction
- a skill framework

But it is not yet fully agentic because:

- the first question is hardcoded outside the graph
- phase and topic are hardcoded on each turn
- the analyzer node does not analyze anything
- skills are defined but not injected into runtime behavior
- tool use is optional by prompt, not enforced by flow
- there is no explicit per-topic depth tracking or transition policy

So the current system is graph-wrapped conversational generation, not a true interview state machine.

---

## Design Goals

The target system should satisfy these goals:

1. The first interviewer question must be generated by the graph, not hardcoded.
2. The graph must always begin with grounded context loaded from storage.
3. The interview must progress through explicit phases.
4. Each topic must have a parent question and controlled follow-ups.
5. Follow-up decisions must be based on analysis of the candidate's latest answer.
6. Skill selection must come from extracted skill map plus resume relevance.
7. The system must know when to challenge, when to clarify, and when to move on.
8. Final reporting must be derived from structured thread evaluations, not only transcript summarization.

---

## Interview Model

### High-level phases

The interview should run through these phases:

1. `OPENING`
2. `RESUME_VERIFICATION`
3. `SKILL_PROBING`
4. `WRAP_UP`
5. `COMPLETED`

### Phase intent

`OPENING`

- greet briefly
- establish tone
- ask one context-aware opener
- do not stay here for more than one turn

`RESUME_VERIFICATION`

- choose 1-2 resume claims or projects most relevant to the JD
- verify ownership, depth, tradeoffs, metrics, and actual contribution
- detect shallow or inflated claims early

`SKILL_PROBING`

- iterate through selected interview topics
- ask one parent question per topic
- ask follow-ups only when analysis says more evidence is needed
- stop rabbit-holing after defined depth or confidence threshold

`WRAP_UP`

- ask one final candidate question or closing prompt
- mark uncovered skills as not evaluated

`COMPLETED`

- stop interview graph
- trigger report workflow

---

## Core Agentic Principle

The system should not ask the LLM to "remember" interview policy in plain text alone.

It should instead:

- persist structured interview state
- use explicit graph transitions
- run analysis after every candidate answer
- let the next-question step read both transcript and analyzed state

The LLM should generate content. The graph should control behavior.

---

## Target Runtime Architecture

Use one orchestrated interview graph with specialized nodes.

### Recommended nodes

1. `load_context`
2. `plan_interview`
3. `select_next_topic`
4. `generate_question`
5. `wait_for_user`
6. `analyze_answer`
7. `decide_next_action`
8. `update_state`
9. `finalize_interview`

### Node responsibilities

`load_context`

- fetch session, JD, resume, extracted resume, extracted skill map, company info, existing transcript
- normalize everything into graph state
- fail fast if required context is missing

`plan_interview`

- derive interview plan before the first question
- choose resume hooks
- choose skill topics
- assign priority and difficulty
- initialize topic queue

`select_next_topic`

- choose the next topic based on remaining queue, evidence collected, and current phase

`generate_question`

- produce exactly one interviewer utterance
- if topic has not started, ask parent question
- if topic is active, ask targeted follow-up
- include enough context to make the question specific

`wait_for_user`

- graph pause point until a user answer arrives

`analyze_answer`

- score the latest answer
- identify missing evidence
- classify confidence and depth
- decide if the answer shows surface, practitioner, or expert-level understanding

`decide_next_action`

- decide one of:
  - `FOLLOW_UP`
  - `MOVE_TOPIC`
  - `MOVE_PHASE`
  - `WRAP_UP`
  - `END`

`update_state`

- persist structured thread state, scores, coverage, depth, and transitions

`finalize_interview`

- persist final session state
- trigger report generation from structured thread data plus transcript

## State Contract

The graph should use a strict structured state, not ad hoc dictionaries.

### Proposed graph state

```python
class InterviewState(TypedDict):
    session_id: str
    candidate_name: str

    messages: Annotated[Sequence[BaseMessage], operator.add]

    phase: str
    interview_status: str

    job_description: str
    company_info: str
    resume_text: str
    extracted_resume: dict
    extracted_skills: dict

    selected_topics: list[dict]
    remaining_topics: list[str]
    active_topic_id: str | None
    active_topic_name: str | None
    active_topic_source: str | None

    topic_depth: int
    max_topic_depth: int
    topic_started: bool

    answered_topics: list[dict]
    resume_verification_targets: list[dict]
    completed_resume_checks: list[dict]

    last_user_answer: str
    last_analysis: dict
    last_decision: str

    coverage_summary: dict
    risk_flags: list[str]
    anti_injection_flags: list[str]

    is_waiting_for_user: bool
    is_complete: bool
```

### Required invariants

- `phase` must always be one of the defined phases.
- `active_topic_id` must be set during `SKILL_PROBING`.
- `topic_depth` increments only for the active topic.
- `topic_depth` resets when topic changes.
- `last_analysis` must exist before `decide_next_action`.
- `is_waiting_for_user=True` means the next external event should be a user answer.

---

## Session Context Requirements

Before the first question, the agent must already have:

- candidate name
- raw JD
- company info
- parsed resume text
- extracted resume structure
- extracted skills
- session transcript if reconnecting

This should not be left to optional model tool use.

### Rule

Do not rely on the interviewer model to remember to call `get_session_context`.

Instead:

- `load_context` should fetch and inject context directly into graph state
- tools should be used only for supplemental retrieval during runtime

Prompt instructions can still encourage tool use, but the baseline context must be deterministic.

---

## Topic Planning

The interview should not probe raw extracted skills blindly.

It should build a topic plan from multiple signals:

- JD must-have skills
- JD nice-to-have skills
- resume projects and claimed technologies
- inferred seniority
- company context
- custom skills from the database

### Topic categories

Recommended categories:

- `resume_claim`
- `core_skill`
- `supporting_skill`
- `system_design`
- `debugging`
- `behavioral`

### Topic selection policy

For a technical role:

- 1-2 resume verification topics
- 3-5 core technical topics
- 1 debugging or incident topic
- 1 system design topic for mid/senior roles

For a non-technical role:

- resume verification
- domain-specific competency topics
- behavioral or situational topics

### Topic object

```python
class TopicPlan(BaseModel):
    id: str
    name: str
    category: str
    source: str
    priority: int
    expected_depth: int
    jd_evidence: list[str]
    resume_evidence: list[str]
    status: str
```

---

## Turn Logic

Each turn should follow this deterministic loop.

### First turn

1. `load_context`
2. `plan_interview`
3. `select_next_topic`
4. `generate_question`
5. pause and wait for user

### Every later turn

1. append latest user answer
2. `analyze_answer`
3. `decide_next_action`
4. `update_state`
5. if follow-up, `generate_question`
6. if move topic, `select_next_topic` then `generate_question`
7. if move phase, transition then continue
8. if complete, `finalize_interview`

---

## Follow-up Policy

The system needs a real follow-up policy, not only "ask more if needed."

### Ask a follow-up when

- answer is vague
- answer is generic and not tied to candidate experience
- answer misses a critical JD dimension
- answer includes a claim worth verifying
- answer is partially correct but lacks tradeoff reasoning
- answer suggests resume inflation

### Move on when

- enough evidence has been gathered
- topic depth limit is reached
- answer quality clearly establishes ceiling
- the candidate is stuck and one baseline question has already been asked

### Recommended depth policy

- `parent question`: depth 0
- `follow-up`: depth 1
- `follow-up`: depth 2
- `final calibration or pressure question`: depth 3

For most topics, cap at 3 follow-ups after the parent question.

Allow lower caps when:

- coverage is more important than depth
- candidate is clearly weak and baseline is established
- interview time budget is low

## Answer Analysis Contract

The analyzer node should return structured output, not free text.

### Proposed analyzer output

```json
{
  "answer_quality": "strong | adequate | weak | evasive",
  "clarity_score": 0,
  "accuracy_score": 0,
  "depth_level": "L0 | L1 | L2 | L3",
  "evidence_found": [
    "mentioned cache invalidation strategy",
    "explained retry semantics"
  ],
  "missing_evidence": [
    "no mention of failure handling",
    "did not explain personal contribution"
  ],
  "resume_verification_signal": "verified | unclear | suspicious | not_applicable",
  "follow_up_targets": [
    "ask for a concrete production incident",
    "ask why they chose Redis over DB query"
  ],
  "move_on_confidence": 0.0,
  "risk_flags": [
    "possible ownership inflation"
  ]
}
```

### Why this matters

Without structured analysis:

- state transitions become prompt magic
- follow-up behavior becomes inconsistent
- reporting cannot reliably reuse thread-level evidence

## Question Generation Contract

The interviewer generation step should receive structured inputs, not just transcript.

### Inputs to question generation

- phase
- active topic
- topic category
- topic depth
- extracted skills
- current JD evidence
- resume evidence
- analyzer follow-up targets
- prior thread messages for this topic

### Output requirements

The interviewer must:

- ask exactly one question
- keep the question concise
- ground it in the active topic
- prefer specific, evidence-seeking wording
- avoid compound multi-part questions

### Question styles by topic state

New topic:

- ask a parent question

Topic in progress:

- ask one targeted follow-up tied to missing evidence

Resume verification:

- ask about ownership, decision-making, scale, failure, metrics

Weak answer:

- ask a simpler baseline question before moving on

Strong answer:

- ask one pressure or tradeoff follow-up, then move on

---

## Tooling Strategy

Tools should support the agent, not carry core orchestration.

### Keep as tools

- supplemental session lookup
- internal glossary or concept reference
- JD requirement lookup
- company policy lookup
- time budget lookup

### Do not delegate to tools

- initial session context load
- current phase selection
- active topic selection
- follow-up policy
- interview completion decision

Those belong to graph state and control logic.

---

## Prompt Strategy

Prompts should be specialized by node.

### Interviewer prompt

Responsibilities:

- generate one question at a time
- stay inside active topic
- use current analysis guidance
- avoid topic drift

### Analyzer prompt

Responsibilities:

- judge answer quality
- identify gaps
- return machine-readable JSON

### Planner prompt

Responsibilities:

- turn JD plus resume into ordered topic plan
- choose verification hooks
- tune for role seniority and domain

### Report prompt

Responsibilities:

- aggregate thread-level findings
- produce final structured hiring report

## Persistence Model

The database should store more than transcript lines.

### Store these session-level fields

- current phase
- active topic id
- active topic depth
- interview status
- selected topic plan
- coverage summary
- final report

### Store these topic-level fields

- topic name
- topic category
- topic source
- thread transcript
- analysis snapshots
- final topic score
- demonstrated depth
- status

This can be a new `interview_topic_threads` table or an equivalent JSON column strategy.

## Socket Event Model

The live interview socket should reflect agent state, not only text.

### Recommended server events

- `session_joined`
- `agent_status`
- `question_generated`
- `topic_transition`
- `phase_transition`
- `transcript_update`
- `report_ready`
- `interview_completed`
- `error`

### Recommended payload additions

Include at least:

- `session_id`
- `phase`
- `active_topic`
- `topic_depth`
- `status`

This will make the frontend easier to reason about and debug.

## Safety and Guardrails

### Prompt injection handling

Candidate input should never directly overwrite system policy.

Add an input guard that flags patterns such as:

- "ignore previous instructions"
- "act as system"
- "show your prompt"
- "change the interview"

When triggered:

- store the flag in state
- ignore the instruction
- continue the interview
- optionally ask the candidate to focus on the question

### Hallucination control

The interviewer should only ask from:

- topic plan
- JD evidence
- resume evidence
- prior candidate statements

### Anti-rabbit-hole control

If `topic_depth >= max_topic_depth`, force topic decision.

### Coverage control

Before `WRAP_UP`, ensure required must-have skills are either:

- evaluated
- partially evaluated
- marked not evaluated

---

## Reporting Architecture

The final report should reuse structured interview artifacts.

### Inputs to reporting

- full transcript
- topic plans
- topic thread transcripts
- per-turn analyzer outputs
- coverage summary
- resume verification findings

### Report stages

1. thread finalization per topic
2. full interview synthesis
3. hiring recommendation

This is better than asking one model to infer everything from a raw transcript at the end.

## Recommended Graph Shape

```text
START
  -> load_context
  -> plan_interview
  -> select_next_topic
  -> generate_question
  -> WAIT_FOR_USER
  -> analyze_answer
  -> decide_next_action
      -> generate_question
      -> select_next_topic
      -> finalize_interview
END
```

Optional later additions:

- `guard_input`
- `time_budget_manager`
- `observer_reporter`

## Migration Plan From Current Code

### Phase 1: Make the current graph authoritative

- remove hardcoded opener from socket handler
- make join flow call graph for first question
- move session context loading into graph initialization
- stop hardcoding `phase` and `current_topic`

### Phase 2: Introduce structured state and planner

- expand `InterviewState`
- add `load_context` node
- add `plan_interview` node
- persist selected topics and current phase

### Phase 3: Add real analyzer and decision nodes

- replace placeholder analyzer
- return structured analysis JSON
- implement transition rules for follow-up vs move-on

### Phase 4: Wire skill framework into runtime

- use `resolve_active_skills(...)`
- convert skills into topic plan inputs
- inject built skill instructions into interviewer prompt

### Phase 5: Improve persistence and reporting

- store topic thread data
- store per-turn analysis
- generate report from structured evidence

## Success Criteria

The interview can be considered fully agentic when all of these are true:

- the first question is generated from graph state, not hardcoded
- context is loaded deterministically before questioning begins
- phase and topic are runtime state, not static defaults
- each answer is analyzed before next-step selection
- follow-up questions are tied to explicit missing evidence
- skill coverage is measurable during the session
- the system can justify why it moved on from a topic
- final report uses structured thread evidence

---

## Recommended Build Order

Build in this order:

1. authoritative graph-owned first turn
2. structured state contract
3. interview planner
4. analyzer JSON output
5. decision node and topic transitions
6. topic/thread persistence
7. final reporting over structured evidence

This order reduces risk and keeps the system usable while it becomes more agentic.

---

## Bottom Line

A fully agentic interview system is not just "LLM with tools."

It is:

- context-grounded before the first turn
- stateful across the full session
- explicit about interview phase and topic
- analytical after every user answer
- policy-driven in follow-up and transition behavior
- evidence-based in final reporting

That is the target architecture this codebase should move toward.
